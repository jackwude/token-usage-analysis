# MEMORY.md - 长期记忆

## 用户偏好

- **工具策略**：启用「技能优先模式」——只要存在明确匹配的已安装 Skill，就优先调用 Skill；若我不确定是否有匹配 Skill 或匹配度不高，我会**先试最可能的 Skill**（更快），失败/不通再向你确认。
- **决策选项格式**：暂时**不要**使用 Telegram 内联按钮（inline buttons）。需要用户决策时，直接在正文列出 1/2/3… 选项，我用数字标号回复，用户用数字回复结论即可。
- **Context 使用率显示**：仅当 Context 使用率 >50% 时，在每次回复末尾附带 Context 信息（例如 120k/272k | 44%）。
- **对话轮次提醒**：每个会话对话超过 10 轮时，提醒是否需要开启新对话（给出选项：继续此对话 / 开新对话）。
- **车辆信息**：车牌号为 川 A1B0A4。
- **欢迎语风格**：活泼 + 正式融合版。
- **回复长度偏好**：默认使用“标准精简版”回复风格——先给结论，再给少量关键点；整体适度精简、便于快速阅读，不要大段铺陈，除非用户明确要求展开。用户当前有两个 Agent（含 `TG Office`），该偏好应对两个 Agent 同时生效。
- **影视资源搜索**：用户会请求帮忙搜索电影/电视剧/动漫资源，需要在观影网 (gying.net) 上搜索并提供前 10 个**有效**百度网盘链接（标题 + 链接 + 豆瓣评分）。
- **快捷命令**：
  - "影视搜索：XXX" → 调用 gying-search Skill (v0.4)，搜索 XXX 的网盘资源（含豆瓣评分，优先最新资源）
  - "gying 搜索：XXX" → 同上
  - "找一下 XXX 的网盘资源" → 同上
  - "帮我搜 XXX，要带评分" → 同上
  - "查一下百炼套餐用量" / "看看阿里云还剩多少额度" / "百炼用量情况" / "查百炼额度" / "百炼用量" → 调用 `bailian-usage` Skill，通过浏览器登录阿里云百炼控制台，查询 Coding Plan 套餐用量和有效期（套餐状态、剩余天数、用量消耗百分比、刷新时间）
  - "查 Token" / "Token 用量" / "Token 消耗" → **仅查 OpenClaw/阿里云百炼 API 的 Token 用量**，不管 codexbar（Cursor 编辑器）的 Token
- **Skill 执行透明度**：执行任何 Skill 时，必须主动告知用户正在使用的是哪个 Skill（例如"📚 正在使用 `gying-search` Skill..."），保持操作透明。
- **百炼用量回复格式**：精简版，去掉 API 配置信息，但保留用量消耗的刷新时间。
- **搜索工具优先级**：信息查询任务默认“自动选择”——优先用 `tavily-skill` 做实时搜索；如果 Tavily 额度/Key 不可用或报配额耗尽，则自动切换到 `mcporter + Exa MCP`（Agent Reach 的全网语义搜索）；只需读单页内容时用 `web_fetch`；需要交互操作/登录时再用 `browser`。
- **凌晨任务确认**：凌晨时段（00:00-07:00）收到任务时，主动询问是「今天」还是「明天」执行，避免时间歧义。

## 其他记忆

- **长任务三段式协议（方案 1）**：任何预计 >15 秒的任务，必须：
  1) **开始即回**：说明在做什么 + 预计耗时区间 + 进度播报频率（默认每 2-3 分钟）。
  2) **过程中不静默**：每 2-3 分钟主动更新一次（或每完成 25% 更新）。
  3) **结束必回**：无论成功/失败都要推送结果；失败要说明原因 + 已尝试 + 下一步。
- **最小看门狗兜底（方案 2）**：长命令优先使用 `scripts/longtask-run.sh` 包装运行（interval 默认 150s），通过周期性 heartbeat 防止“无输出导致看起来挂了”。（实现说明见 `scripts/LONGTASKS.md`）
- **即时响应用户消息**：即使后台任务未完成，用户发消息必须立即回复，不能让用户体验到“挂起”。
- **浏览器资源策略**：每次完成浏览器相关任务后，默认关闭所有非必要 tab/页面（激进省内存），避免长期堆积导致卡顿；仅在明确需要保留现场时才保留。
- **浏览器任务收尾习惯**：每次执行完浏览器相关任务后，要主动自行关闭本次打开的页面，避免内存占用过高；除非用户明确要求保留现场。
- **openclaw.json 修改协议**：修改前必须：① 先评估报错风险 ② 第一次确认（说明修改内容和原因）③ 第二次确认（用户再次同意）④ 最后才执行修改。
- **文件删除/破坏性操作协议（用户确认启用）**：任何删除/覆盖操作必须满足：① 用户提供完整路径点名要操作的文件/目录 ② 我复述将要执行的具体命令与影响 ③ 用户二次确认后才执行；默认优先“移到废纸篓/备份/改名”，避免不可恢复删除。
- **长任务防卡协议**：所有耗时任务（>30 秒）必须：
  1. 优先 spawn sub-agent 后台执行，保持主会话响应
  2. 严格三段式：开始时告知「任务 + 预计时长 + 汇报频率」→ 过程中每 2-3 分钟更新 → 结束必回（成功/失败）
  3. 用户发消息必须立即回复，不能让用户体验到"挂起"
  4. 浏览器任务完成后清理非必要 tab
