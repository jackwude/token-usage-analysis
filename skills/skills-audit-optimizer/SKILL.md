# skills 审计优化工具

审计、评分、诊断并优化已有 skill。

## 核心定位

这个 skill 不直接完成业务任务，而是专门回答这些上位问题：

- 这个 skill 设计得是否合理
- 当前短板在哪
- 应该优先改哪里
- 该怎么改，收益最大

把它当成一个面向 skill 的“审计器 + 优化器”。

## 什么时候用

适用于：

- 发布前体检某个 skill
- 审计一个已安装或已编写的 skill
- 比较多个 skill 的质量和短板
- 给某个 skill 生成优先级明确的优化建议
- 基于真实使用记录，复盘某个 skill 的表现

不适用于：

- 从零设计业务型 skill 的具体内容
- 只想直接改一个 skill、但不需要先审计判断

## 工作原则

- 先给总体判断，再展开关键问题。
- 不凭感觉打分；每个分数都要有依据。
- 优先找主要矛盾，不要把所有小问题一股脑堆出来。
- 建议必须可执行，能落到具体文件位置、具体改法、具体收益。
- 有真实使用记录时，优先结合真实案例审计；不要只盯着 SKILL.md 静态文本。

## 审计流程

按下面顺序执行，除非用户明确要求只做其中一部分。

### 1. 确认审计对象和范围

先确认：

1. 审计对象是哪个 skill
2. 审计范围是静态文件、真实案例、还是两者结合
3. 审计目标是发布前体检、质量评分、找短板、给优化建议，还是横向比较多个 skill

如果用户没说清，先用最少的问题补齐必要信息。

### 2. 选择审计模式

根据用户目标和材料完整度，选择一种模式：

- **快速审计**：只给结论、最大短板、1~3 条建议
- **完整审计**：按四维量表完整评分并输出正式报告
- **对比审计**：对多个 skill 使用同一量表后横向比较

需要细则时，读取 `references/modes.md`。

### 3. 读取 skill 本体

至少检查：

- `SKILL.md`
- `scripts/`（如果有）
- `references/`（如果有）
- `assets/`（如果有）

重点关注：

- frontmatter 的 `name` 和 `description`
- SKILL.md 是否清楚描述工作流
- 是否存在信息混堆、重复、失焦
- 是否把高频稳定动作沉淀成 scripts
- 是否把细节资料合理拆进 references

### 4. 如有真实样本，结合动态表现审计

优先看：

- 该触发时有没有触发
- 是否误触发
- 流程是否跑偏、跳步、漏步
- 输出是否稳定
- 用户是否真正获得价值

如果样本过少，明确说明结论置信度有限。

### 5. 按四维量表评分

固定从下面四个一级维度评分：

1. 触发质量
2. 流程质量
3. 结果质量
4. 结构质量

完整量表读取 `references/audit-rubric.md`。

### 6. 提炼关键问题并给出动作建议

只提炼 1~3 个真正影响效果的核心问题。

建议按优先级输出：

- **P0**：必须先修，不修会明显影响使用
- **P1**：修了会显著变好
- **P2**：锦上添花

每条建议都写成动作：

- 修改位置
- 当前问题
- 建议动作
- 预期收益

正式输出模板读取 `references/output-template.md`。

## 默认输出要求

- 用户只要结论时，用轻量输出。
- 用户要正式评审时，用完整审计报告。
- 用户要比较多个 skill 时，优先比较四维平均分、核心短板、P0 问题，不要只看总分。
- 任一一级维度平均分低于 2.5，直接列为重点整改项。

## 不要这样做

- 不要只根据印象下结论。
- 不要把所有小问题堆成很长清单。
- 不要只说“建议更清晰一点”，必须说清楚改哪里、怎么改。
- 不要只读 SKILL.md 就武断下结论；有真实案例时要优先结合真实表现。

## 适合追加的专项维度

遇到特定类型 skill 时，可在四维框架之外追加专项审计：

- **安全性**：涉及外部 API、自动化登录、发消息、文件覆盖、删除操作
- **脚本可靠性**：skill 内含 scripts 且依赖执行结果
- **领域适配度**：面向特定行业、业务规则、专业术语
- **外部依赖稳定性**：依赖网页结构、账号状态、API 配额、第三方平台

如无特别需要，不要一开始就无限扩展维度；先用四维主框架完成基础审计。
