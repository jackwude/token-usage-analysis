# 审计量表

按四个一级维度审计：触发质量、流程质量、结果质量、结构质量。

每个二级项按 1~5 分评分：
- 1 分：严重不足
- 2 分：勉强可用
- 3 分：基本合格
- 4 分：比较成熟
- 5 分：优秀

---

## 一、触发质量

### 1.1 任务定义清晰度
检查：
- skill 到底做什么，是否一句话能说清
- description 是否表达清楚能力边界
- 是否避免过度泛化

### 1.2 触发场景覆盖度
检查：
- 是否写清什么时候用
- 是否覆盖常见用户表达
- 是否覆盖核心触发语义

### 1.3 误触发风险控制
检查：
- 是否容易和其他 skill 冲突
- description 是否过宽
- 是否写清不适用场景

### 1.4 触发后的预期一致性
检查：
- description 承诺的内容，body 是否真能支持
- 用户预期和 skill 实际能力是否一致

---

## 二、流程质量

### 2.1 主流程完整性
检查：
- 是否有清晰主流程
- 是否缺关键步骤
- 步骤之间是否成链条

### 2.2 分支判断清晰度
检查：
- 不同情境下是否有不同处理路径
- 分支条件是否明确
- 是否大量依赖临场猜测

### 2.3 指令可执行性
检查：
- 指令是具体动作还是空泛建议
- Agent 看完后能不能直接执行
- 是否存在模糊表述

### 2.4 异常与失败处理
检查：
- 出错时怎么办
- 输入不足时怎么办
- 前置条件不满足时怎么办
- 是否有兜底策略

---

## 三、结果质量

### 3.1 输出目标明确度
检查：
- 最终要产出什么是否清楚
- 用户能否判断任务是否完成

### 3.2 输出格式规范度
检查：
- 是否规定输出结构或模板
- 是否容易漏关键项
- 不同执行者输出差异是否过大

### 3.3 结果稳定性
检查：
- 同类任务多次执行是否结果接近
- 是否过度依赖临场发挥
- 是否容易忽好忽坏

### 3.4 用户价值密度
检查：
- 结果是否真正解决问题
- 是否信息过多、过少、过散
- 是否能帮助用户直接决策或进入下一步

---

## 四、结构质量

### 4.1 信息分层合理性
检查：
- description / SKILL.md / references / scripts / assets 的分工是否合理
- 是否把所有信息都堆在一起

### 4.2 上下文效率
检查：
- SKILL.md 是否过长
- 是否充满非必要内容
- 是否遵循 progressive disclosure

### 4.3 可维护性
检查：
- 后续修改是否方便
- 是否有重复内容导致牵一发动全身
- 新增能力是否易扩展

### 4.4 资源组织成熟度
检查：
- scripts / references / assets 是否使用合理
- 有没有该拆不拆、该留不留的问题

---

## 使用规则

- 任一一级维度平均分低于 2.5，直接列为重点整改项。
- 不要只看总分；短板维度通常比平均分更重要。
- 如果是多个 skill 比较，优先比较四维平均分、核心短板、P0 问题。
